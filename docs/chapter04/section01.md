## 为什么需要国内替代方案？

在我们的项目中，最初计划使用GitHub作为代码托管和发布平台，但考虑到国内的网络环境，我们需要选择更稳定的国内替代方案。

### GitHub在国内的挑战：
- ⚠️ **访问不稳定**：由于网络原因，GitHub在国内的访问速度较慢，有时甚至无法访问
- ⚠️ **加载时间长**：网页加载时间长，影响用户体验
- ⚠️ **功能受限**：某些功能可能无法正常使用

### 国内替代方案的优势：
- 🚀 **访问速度快**：国内平台的访问速度明显更快
- 🛡️ **稳定性高**：不受国际网络波动的影响
- 📍 **本地化服务**：提供更适合国内用户的服务和支持

---

## 国内替代方案评估

让我们系统地评估几个主要的国内替代方案：

### 1. 代码托管平台

| 平台 | 优势 | 劣势 |
|------|------|------|
| **Gitee** | 完全国内托管，访问速度快，界面与GitHub类似 | 部分高级功能需要付费，社区规模相对较小 |
| **Coding.net** | 提供代码托管和项目管理服务 | 免费版功能有限 |
| **阿里云Code** | 阿里云提供的代码托管服务，稳定性高 | 集成度较高，适合阿里云用户 |

### 2. 静态网站托管平台

| 平台 | 优势 | 劣势 |
|------|------|------|
| **Cloudflare Pages** | 全球CDN加速，国内访问速度快，免费托管静态网站 | 需要连接到GitHub或Gitee等代码仓库 |
| **Vercel** | 部署速度快，支持自动部署 | 国内访问速度不稳定 |
| **Netlify** | 功能丰富，支持自动部署 | 国内访问速度不稳定 |
| **GitHub Pages** | 与GitHub无缝集成 | 国内访问速度慢 |

## 我们的选择：Cloud Native Build（cnb.cool）

经过评估，我们选择了**Cloud Native Build（cnb.cool）**作为一站式解决方案，原因如下：

### 为什么选择Cloud Native Build？
- **一站式服务**：同时提供代码托管和静态网站托管功能
- **国内服务**：完全国内托管，访问速度快，不受国际网络影响
- **功能完整**：提供类似GitHub的代码托管功能，包括代码管理、Issues、Pull requests等
- **易于使用**：操作流程简单直观，界面友好
- **成本低**：基本功能完全免费

### 具体优势
1. **稳定性**：国内访问稳定可靠
2. **速度快**：国内服务器确保快速加载
3. **易于使用**：操作流程简单直观
4. **成本低**：基本功能完全免费
5. **功能完整**：支持静态网站的所有必要功能
6. **国内服务**：完全国内托管，不受国际网络影响
7. **一站式服务**：同时提供代码托管和网站部署功能

## 步骤1：注册Cloud Native Build账号

### 操作步骤：

1. **访问Cloud Native Build官网**
   - 打开浏览器，访问 [https://cnb.cool/](https://cnb.cool/)

2. **注册账号**
   - 点击「注册」按钮
   - 输入邮箱和密码
   - 完成注册

3. **登录账号**
   - 使用注册的账号和密码登录

---

## 步骤5：配置Cloud Native Build

### 操作步骤：

1. **进入项目管理**
   - 登录后，进入控制台
   - 点击「创建项目」按钮

2. **创建新项目**
   - **项目名称**：输入 `tutorial-project`
   - **描述**：输入「使用Solo Trae Web Code制作的网页教程项目」
   - **仓库类型**：选择「公开」
   - 点击「创建」按钮

3. **上传项目文件**
   - 进入项目页面，点击「上传文件」按钮
   - 上传项目的所有文件，包括：
     - `index.html`
     - `css/style.css`
     - `js/script.js`
   - 或者使用Git推送代码到Cloud Native Build仓库

4. **配置构建设置**
   - 进入项目设置
   - **构建目录**：输入 `project`（因为我们的网页文件在project目录中）

5. **部署项目**
   - 点击「部署」按钮
   - 等待部署完成

---

## 步骤6：访问部署的网站

### 操作步骤：

1. **获取网站URL**
   - 部署完成后，Cloud Native Build会生成一个唯一的URL
   - 格式通常为：`https://cnb.cool/<username>/<project-name>`

2. **访问网站**
   - 打开浏览器，输入生成的URL
   - 查看网站是否正常加载

3. **验证网站内容**
   - 确保所有页面和资源都能正常访问
   - 验证样式和脚本是否正常工作

### 实际部署示例：

我们的项目已成功部署到：
- **部署地址**：[https://cnb.cool/slowdown87/tutorial-project](https://cnb.cool/slowdown87/tutorial-project)
- **访问方式**：直接在浏览器中打开上述地址

## 国内替代方案使用最佳实践

### 1. Gitee使用技巧

## 常见问题与解答

### Q: Gitee和GitHub有什么区别？
**A:** Gitee是国内的代码托管平台，访问速度快，适合国内用户；GitHub是国际平台，社区更活跃，资源更丰富。

### Q: Cloud Native Build在国内访问速度如何？
**A:** Cloud Native Build是国内服务，访问速度非常快，完全不受国际网络影响。

### Q: 如何配置自定义域名？
**A:** 在Cloud Native Build的项目设置中，添加自定义域名，然后在域名提供商处配置DNS记录。

### Q: 部署失败怎么办？
**A:** 检查构建配置是否正确，特别是构建目录是否指向正确的目录。查看部署日志，了解具体的错误信息。

### Q: 如何回滚到之前的版本？
**A:** 在Cloud Native Build的部署历史中，找到之前的成功部署，点击「回滚」按钮。
